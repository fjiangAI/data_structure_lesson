# Week 05 受限线性表之二：队列 练习题

## A. 基础必做：概念与结构

1. 用不超过 120 字说明本周结构保存了什么“对象”和什么“关系”。
2. 画出一个包含 5 个元素的小例子，标出关键字段、指针、下标、父子关系或邻接关系。
3. 写出本周最核心 ADT 的 C 风格接口，至少包含初始化、核心操作、销毁或清理。
4. 解释本周结构在课程主线中的位置：它相对于上一类结构增加了什么表达能力，付出了什么代价？

## B. 手推必做：操作过程

5. 手工执行 `enqueue`，逐步写出每一步读了哪些字段、写了哪些字段。
6. 对 `dequeue` 给出正常路径和失败路径各一个例子。
7. 选择一个操作，写出“操作前不变量、操作步骤、操作后不变量仍成立”的三段式说明。

## C. 编程必做：代码阅读与测试

8. 阅读 [examples/circular_queue.c](examples/circular_queue.c)，指出结构体定义、初始化函数、核心修改函数和输出函数。
9. 给示例代码补 3 个测试：正常情况、边界情况、错误输入或失败路径。
10. 找出示例代码中最依赖 C 语言细节的一处，例如指针、数组下标、动态内存或字符串结束符。

## D. 复杂度与证明（挑战）

11. 对下列操作分别说明输入规模和复杂度来源：

    - enqueue：O(1)，关键动作：从 rear 位置加入元素
    - dequeue：O(1)，关键动作：从 front 位置取出元素
    - front：O(1)，关键动作：读取队头元素
    - isFull：O(1)，关键动作：(rear + 1) % capacity == front
    - 层序处理：O(n)，关键动作：当前对象出队，其相邻对象入队

12. 构造一个最坏情况输入，并说明它为什么触发最坏复杂度。
13. 如果输入规模扩大 100 倍，本周结构最先暴露的瓶颈是什么？

## E. 错误代码诊断（必做）

下面这段代码很像 LLM 生成的简化实现，请指出 bug、给出触发 bug 的输入，并写出修正思路。

```c
return q->front == q->rear;
```

诊断要求：

1. 说明它破坏了哪个结构不变量。
2. 写一个最小反例。
3. 给出修正后的关键语句或伪代码。

## F. 本周专属训练

1. 手推 8 次入队/出队混合操作
2. 比较牺牲一个单元和增加 size 字段两种方案
3. 设计队满后继续入队测试

## G. 设计与 AI 审查（拓展）

1. 比较两种循环队列设计：牺牲一个单元 vs 维护 `size` 字段。说明各自的队满判定、不变量和易错点。
2. 构造一个能暴露 `rear` 不取模或队满判定错误的入队/出队序列。
3. 让 LLM 生成循环队列代码，检查它是否混淆 `front` 指向队头还是空位。
4. 提交证据：圆环状态图、错误判定公式、修正后的空满判断。
