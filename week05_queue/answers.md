# Week 05 受限线性表之二：队列 参考答案

> 参考答案用于校准思路，不要求学生逐字一致。程序题重点看接口、边界、复杂度说明和测试证据。

## A. 基础概念参考

1. 本周组织形式应围绕“元素如何保存关系”回答。例如：队列在队尾入队、队头出队。循环队列用取模把数组首尾连接起来，避免普通数组队列的整体搬移。
2. 图示答案应包含元素、位置或指针关系，并标出操作会改变的位置。树或图题需要标出根、孩子、顶点、边或邻接关系。
3. 函数原型示例可以写成 `int operation(Structure *s, ElemType value);`。关键是区分输入、输出、结构指针和失败返回值。
4. 本周在课程主线中的位置可以概括为：限制不是削弱，而是换来更强的不变量：最近未完成、先来先服务、已匹配前后缀。

## B. 操作推演参考

5. `enqueue` 的关键动作是：从 rear 位置加入元素。推演时必须写出读字段、写字段和不变量。
6. 失败路径至少包含空结构、满结构、越界、查找失败、重复关键字或非法输入中的一种。
7. 三段式证明示例：操作前结构满足 队列 的定义；操作中只修改必要字段；操作后对象关系仍然符合定义。

## C. 代码阅读与测试参考

8. 初始化函数负责建立合法空状态；核心修改函数负责改变结构字段；输出函数用于观察状态，不应替代测试。
9. 合格测试至少包含正常路径、边界路径和失败路径。例如空结构删除应失败，查找不存在元素应返回 -1 或 NULL，容量不足应扩容或拒绝。
10. C 语言细节通常落在数组下标、指针重连、动态内存、字符串结束符、递归栈或结构体字段初始化。

## D. 复杂度或设计评审参考

- enqueue：O(1)。推导依据：从 rear 位置加入元素。
- dequeue：O(1)。推导依据：从 front 位置取出元素。
- front：O(1)。推导依据：读取队头元素。
- isFull：O(1)。推导依据：(rear + 1) % capacity == front。
- 层序处理：O(n)。推导依据：当前对象出队，其相邻对象入队。

最坏情况需要明确输入规模。例如元素个数 `n`、顶点数 `V`、边数 `E`、模式串长度 `m` 或关键字范围 `k`。不能只写大 O 结论。

## E. 错误代码诊断参考

题目：循环队列队满判断错误

代码：

```c
return q->front == q->rear;
```

问题：front==rear 表示队空；若牺牲一个单元，队满应为 (rear+1)%cap==front。

评分时重点看学生是否能说出：

1. 哪个不变量被破坏。
2. 哪个最小输入可以触发错误。
3. 修正方案是否真的覆盖边界。

## F. 本周专属训练参考

本周专属训练重点看学生是否完成了下面这些能力：

- 手推 8 次入队/出队混合操作
- 比较牺牲一个单元和增加 size 字段两种方案
- 设计队满后继续入队测试

参考答案不要求形式一致，但必须体现本周核心抓手：循环队列 `front/rear` 取模与队满判定。

## G. 设计与 AI 审查参考

参考方向：循环队列要明确 `front` 和 `rear` 的含义。牺牲一个单元时队满条件通常为 `(rear + 1) % capacity == front`；维护 `size` 字段时可使用全部空间，但每次入队出队都必须同步更新 `size`。
