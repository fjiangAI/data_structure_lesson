# Week 16 综合设计、复习与 AI 辅助实践 练习题

## A. 基础必做：概念与结构

1. 用不超过 120 字说明本周结构保存了什么“对象”和什么“关系”。
2. 画出一个包含 5 个元素的小例子，标出关键字段、指针、下标、父子关系或邻接关系。
3. 写出本周最核心 ADT 的 C 风格接口，至少包含初始化、核心操作、销毁或清理。
4. 解释本周结构在课程主线中的位置：它相对于上一类结构增加了什么表达能力，付出了什么代价？

## B. 手推必做：操作过程

5. 手工执行 `需求建模`，逐步写出每一步读了哪些字段、写了哪些字段。
6. 对 `结构选择` 给出正常路径和失败路径各一个例子。
7. 选择一个操作，写出“操作前不变量、操作步骤、操作后不变量仍成立”的三段式说明。

## C. 编程必做：代码阅读与测试

8. 阅读 [examples/student_index_lab.c](examples/student_index_lab.c)，指出结构体定义、初始化函数、核心修改函数和输出函数。
9. 给示例代码补 3 个测试：正常情况、边界情况、错误输入或失败路径。
10. 找出示例代码中最依赖 C 语言细节的一处，例如指针、数组下标、动态内存或字符串结束符。

## D. 复杂度矩阵与设计评审（挑战）

11. 对下列操作分别说明输入规模和复杂度来源：

    - 需求建模：识别对象、关系和高频操作；需要提交 对象-操作表
    - 结构选择：用访问模式决定数组、链表、树、图或哈希；需要提交 结构取舍表
    - 接口实现：保持 ADT 边界清晰；需要提交 头文件与不变量说明
    - 复杂度评审：给出关键操作的上界；需要提交 操作复杂度矩阵
    - AI 辅助验证：让模型生成反例、边界用例和解释；需要提交 反例与修正记录

12. 构造一个最坏情况输入，并说明它为什么触发最坏复杂度。
13. 如果输入规模扩大 100 倍，本周结构最先暴露的瓶颈是什么？

## E. 错误代码诊断（必做）

下面这段代码很像 LLM 生成的简化实现，请指出 bug、给出触发 bug 的输入，并写出修正思路。

```c
list_remove_at(&list, pos); return 1;
```

诊断要求：

1. 说明它破坏了哪个结构不变量。
2. 写一个最小反例。
3. 给出修正后的关键语句或伪代码。

## F. 本周专属训练

1. 设计学生记录系统的 ADT 头文件
2. 列出 add/delete/find/rank 的复杂度矩阵
3. 让 LLM 生成一个删除实现并设计能暴露索引不同步的测试

## G. 设计与 AI 审查（拓展）

1. 为学生记录索引系统写对象-操作-结构选择表，说明顺序表、哈希索引和排名视图各自负责什么。
2. 构造一个删除中间记录后索引不同步的反例，列出修复索引或重建索引的两种方案。
3. 让 LLM 生成 `delete_student`，检查它是否同时维护记录表、学号索引和排名结果。
4. 提交证据：复杂度矩阵、一致性维护清单、能暴露不同步的测试。
