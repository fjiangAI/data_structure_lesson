# Week 04 受限线性表之一：栈 教师教案

## 本周教学意图

本周不是让学生记住 栈 的定义，而是让学生看清它在“点 -> 连续线 -> 不连续线 -> 受限线 -> 树 -> 图 -> 索引 -> 重排 -> 综合系统”这条主线中的位置。

阶段定位：受限线与序列模式

限制不是削弱，而是换来更强的不变量：最近未完成、先来先服务、已匹配前后缀。

## 板书主线

```text
对象：LIFO、栈顶
关系：栈、队列和串仍是线性对象，但通过限制操作端点或强化序列匹配，表达更具体的行动规则。
不变量：栈只允许在栈顶进行插入和删除。限制操作位置带来了简单清晰的状态变化，也让它适合处理最近未完成的任务。
操作：push、pop、peek
代价：从字段读写、循环次数和递归层数推出
```

## 两节课教学流程

### 第 1 节：结构是什么

1. 用问题场景引入：为什么少开放一些操作，反而让算法更容易正确？
2. 画逻辑结构图，只画对象和关系，暂时不写代码。
3. 把逻辑结构翻译为 C 字段，逐个解释字段责任。
4. 让学生说出本周结构的不变量。
5. 使用交互动画走 2 到 4 步，要求学生记录每一步读写了哪些字段。

### 第 2 节：操作如何维护结构

1. 从 C 示例中找到结构体定义、初始化、核心操作和销毁函数。
2. 手推核心操作，要求学生写出失败条件。
3. 根据移动次数、指针跳转次数、递归层数或边扫描次数推导复杂度。
4. 给一段有缺陷的模型生成代码，让学生指出不变量被破坏的位置。
5. 用 1 道随堂题收束：让学生把操作、复杂度和边界条件写在同一张表中。

## 核心操作讲解脚本

1. **push**：元素进入栈顶。分析时先找会重复执行的语句，再判断重复次数，所以代价为 O(1)。
2. **pop**：删除并返回栈顶元素。分析时先找会重复执行的语句，再判断重复次数，所以代价为 O(1)。
3. **peek**：读取栈顶但不删除。分析时先找会重复执行的语句，再判断重复次数，所以代价为 O(1)。
4. **isEmpty**：判断 top 是否为空。分析时先找会重复执行的语句，再判断重复次数，所以代价为 O(1)。
5. **括号匹配**：左括号入栈，右括号匹配栈顶。分析时先找会重复执行的语句，再判断重复次数，所以代价为 O(n)。

## 本周专属讲法

课堂上不要平均用力，建议抓住下面这条线：

1. 扫描 `a[(b+c)*d]`，遇到 `[` 和 `(` 入栈，遇到 `)` 必须匹配栈顶 `(`。
2. 若输入为 `([)]`，读到 `)` 时栈顶是 `[`，类型不匹配，应立即失败。
3. 扫描结束后栈必须为空；否则说明还有未匹配的左括号。

讲解顺序建议：

1. 先问“操作前结构满足什么不变量”。
2. 再问“为了完成操作，必须读哪些字段”。
3. 接着问“写字段的顺序能否交换”。
4. 最后问“操作后如何证明不变量仍然成立”。

## 常见学生误区

- 出栈前不判空
- 只统计左右括号数量，不检查顺序
- 把栈顶下标和元素个数混淆
- 结束时不检查栈是否为空

## 课堂活动

- 3 分钟：学生独立画出 5 个元素的小结构图。
- 5 分钟：两人一组互相检查字段含义。
- 8 分钟：全班一起推演 括号匹配中栈顶保存最近未匹配左括号。
- 6 分钟：让 LLM 生成一个错误版本，学生设计能暴露错误的测试。

## LLM 辅助教学提醒

大模型时代，学习 栈 的目的不是和模型比赛背代码，而是拥有判断模型输出的结构能力。学生需要能看出模型有没有维护 受限线与序列模式 的不变量，能不能处理边界，复杂度前提是否成立，测试是否覆盖了最坏情况。知道得越多，越能把模型当成工具，而不是把模型输出当成事实。

建议教师不要简单禁止学生使用大模型，而是要求他们提交“模型输出审查记录”：模型假设了什么、漏了什么边界、复杂度前提是否正确、自己如何修改。

## 形成性评价

学生本周真正掌握的标志：

- 能用一句话说清 栈 保存了什么关系。
- 能画出操作前后结构图。
- 能写出核心操作的 C 风格接口。
- 能用栈状态解释 LIFO，并处理空栈、类型不匹配和最终非空。
- 能指出一段模型生成代码中最可能出错的地方。
